## Feature 2 — Edit order date (existing order)

v1.0.3 already lets you backdate at *creation* (NewOrder). This adds editing
the date of an order that already exists.

- **OrderDetails.tsx**: add a `<input type="date" max={today}>` bound to the
  date portion of `order.created_at`. Initial value = `order.created_at.slice(0,10)`.
- **saveDetails IPC** (`OrderDetailsInput`): add optional `created_at` (YYYY-MM-DD).
  When present and `/^\d{4}-\d{2}-\d{2}$/`, update `orders.created_at` to
  `${created_at} 12:00:00` (noon, matching the backdate convention). Renderer
  only sends it when the picked date differs from the original date portion.
- **Limitation:** date edit flows through `saveDetails`, which rejects `closed`
  orders. Closed orders cannot have their date changed. Acceptable for now.

## Feature 3 — SOCKS half-step (.5)

Customers sometimes drop a single unmatched sock → 0.5 pair.

- **No DB migration.** `order_garments.quantity` has INTEGER affinity; SQLite
  only narrows REAL→INTEGER when lossless, so 0.5 is stored as REAL. Verified
  behavior, not a schema change.
- **OrderDetails.tsx**: a garment row whose name is `socks` (case-insensitive)
  renders its quantity input with `step=0.5 min=0.5` and no `Math.floor`. All
  other rows keep `step=1` + `Math.floor` (integer).
- **Validation relaxation:**
  - `filledGarments` cell threshold `v.quantity >= 1` → `v.quantity > 0`.
  - `saveDetails` per-garment guard `g.quantity < 1` → `g.quantity <= 0`.

## Feature 4 — Per-section "Other" input

Today there is one shared "Add another clothing type" input at the top; added
names already appear under every wearer section. The aunt wants the input
*inside* each section.

- **OrderDetails.tsx**: move the add-input into each wearer's
  `collapse-content` (bottom of the section). Draft text held in
  `Record<Wearer, string>` so each section has its own field.
- Adding still calls `addExtras` (unchanged), so a type added in one section
  still shows across all sections — examples: headband, wristband, scarf,
  bags, hijab.
- Remove the single top-level input.

## Feature 5 — Iron default price 40, editable

Iron is `pricing='custom'`, `unit='item'`, `default_price=null`.

- **db.ts seed**: change iron seed to `default_price=40`.
- **Migration** (existing DBs): `UPDATE services SET default_price=40 WHERE
  key='iron' AND default_price IS NULL`.
- **OrderDetails** already prefills `unit_price ?? default_price`, so a fresh
  iron item shows 40. Input stays editable because it is `readOnly` only when
  `pricing==='fixed'`; iron is `custom`. No UI change needed.
- dry_clean stays `null` (custom, no default).

## Feature 6 — Urgent surcharge %

Per-order percentage upcharge the aunt types herself (e.g. 50, 100), applied
to the whole order total.

- **DB**: `orders.surcharge_pct REAL NOT NULL DEFAULT 0` via `addColumn`
  migration.
- **types.ts**: `Order.surcharge_pct: number`; `OrderDetailsInput.surcharge_pct: number`.
- **pricing.ts**: `computeOrderTotal(items, surchargePct = 0)` →
  `subtotal * (1 + surchargePct / 100)`. Keeps the function pure.
- **OrderDetails.tsx**: numeric input "Urgent surcharge %" near the total.
  Display breakdown: subtotal, surcharge line (only when pct > 0), grand total.
  Loaded from `order.surcharge_pct` on open.
- **saveDetails IPC**: read `input.surcharge_pct`, validate `>= 0`, compute
  total via `computeOrderTotal`, persist `surcharge_pct` and `total` on the
  order row.
- **Reports / export**: untouched — `orders.total` already includes the
  surcharge.
